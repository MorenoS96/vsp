    <article class="markdown-body entry-content container-lg" itemprop="text"><p dir="auto"><strong>About arc42</strong></p>
<p dir="auto">arc42, the Template for documentation of software and system
architecture.</p>
<p dir="auto">By Dr. Gernot Starke, Dr. Peter Hruschka and contributors.</p>
<p dir="auto">Template Revision: 7.0 EN (based on asciidoc), January 2017</p>
<p dir="auto">© We acknowledge that this document uses material from the arc 42
architecture template, <a href="http://www.arc42.de" rel="nofollow">http://www.arc42.de</a>. Created by Dr. Peter
Hruschka &amp; Dr. Gernot Starke.</p>
<blockquote>
<p dir="auto"><span class="color-fg-accent"><svg class="octicon octicon-info mr-2" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M8 1.5a6.5 6.5 0 100 13 6.5 6.5 0 000-13zM0 8a8 8 0 1116 0A8 8 0 010 8zm6.5-.25A.75.75 0 017.25 7h1a.75.75 0 01.75.75v2.75h.25a.75.75 0 010 1.5h-2a.75.75 0 010-1.5h.25v-2h-.25a.75.75 0 01-.75-.75zM8 6a1 1 0 100-2 1 1 0 000 2z"></path></svg>Note</span></p>

<h1 dir="auto"><a id="user-content-introduction-and-goals" class="anchor" aria-hidden="true" href="#introduction-and-goals"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Introduction and Goals</h1>
<p dir="auto">Describes the relevant requirements and the driving forces that software
architects and development team must consider. These include</p>
<ul dir="auto">
<li>
<p dir="auto">underlying business goals, essential features and functional
requirements for the system</p>
</li>
<li>
<p dir="auto">quality goals for the architecture</p>
</li>
<li>
<p dir="auto">relevant stakeholders and their expectations</p>
</li>
</ul>
<h2 dir="auto"><a id="user-content-requirements-overview" class="anchor" aria-hidden="true" href="#requirements-overview"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Requirements Overview</h2>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">Short description of the functional requirements, driving forces,
extract (or abstract) of requirements. Link to (hopefully existing)
requirements documents (with version number and information where to
find it).</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">From the point of view of the end users a system is created or modified
to improve support of a business activity and/or improve the quality.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">Short textual description, probably in tabular use-case format. If
requirements documents exist this overview should refer to these
documents.</p>
<p dir="auto">Keep these excerpts as short as possible. Balance readability of this
document with potential redundancy w.r.t to requirements documents.</p>
<h2 dir="auto"><a id="user-content-quality-goals" class="anchor" aria-hidden="true" href="#quality-goals"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Quality Goals</h2>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">The top three (max five) quality goals for the architecture whose
fulfillment is of highest importance to the major stakeholders. We
really mean quality goals for the architecture. Don’t confuse them with
project goals. They are not necessarily identical.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">You should know the quality goals of your most important stakeholders,
since they will influence fundamental architectural decisions. Make sure
to be very concrete about these qualities, avoid buzzwords. If you as an
architect do not know how the quality of your work will be judged …</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">A table with quality goals and concrete scenarios, ordered by priorities</p>
<h2 dir="auto"><a id="user-content-stakeholders" class="anchor" aria-hidden="true" href="#stakeholders"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Stakeholders</h2>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">Explicit overview of stakeholders of the system, i.e. all person, roles
or organizations that</p>
<ul dir="auto">
<li>
<p dir="auto">should know the architecture</p>
</li>
<li>
<p dir="auto">have to be convinced of the architecture</p>
</li>
<li>
<p dir="auto">have to work with the architecture or with code</p>
</li>
<li>
<p dir="auto">need the documentation of the architecture for their work</p>
</li>
<li>
<p dir="auto">have to come up with decisions about the system or its development</p>
</li>
</ul>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">You should know all parties involved in development of the system or
affected by the system. Otherwise, you may get nasty surprises later in
the development process. These stakeholders determine the extent and the
level of detail of your work and its results.</p>

<p dir="auto">Table with role names, person names, and their expectations with respect

<h1 dir="auto"><a id="user-content-architecture-constraints" class="anchor" aria-hidden="true" href="#architecture-constraints"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Architecture Constraints</h1>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">Any requirement that constrains software architects in their freedom of
design and implementation decisions or decision about the development
process. These constraints sometimes go beyond individual systems and
are valid for whole organizations and companies.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Architects should know exactly where they are free in their design
decisions and where they must adhere to constraints. Constraints must
always be dealt with; they may be negotiable, though.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">Simple tables of constraints with explanations. If needed you can
subdivide them into technical constraints, organizational and political
constraints and conventions (e.g. programming or versioning guidelines,
documentation or naming conventions)</p>
<h1 dir="auto"><a id="user-content-system-scope-and-context" class="anchor" aria-hidden="true" href="#system-scope-and-context"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>System Scope and Context</h1>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">System scope and context - as the name suggests - delimits your system
(i.e. your scope) from all its communication partners (neighboring
systems and users, i.e. the context of your system). It thereby
specifies the external interfaces.</p>
<p dir="auto">If necessary, differentiate the business context (domain specific inputs
and outputs) from the technical context (channels, protocols, hardware).</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">The domain interfaces and technical interfaces to communication partners
are among your system’s most critical aspects. Make sure that you
completely understand them.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">Various options:</p>
<ul dir="auto">
<li>
<p dir="auto">Context diagrams</p>
</li>
<li>
<p dir="auto">Lists of communication partners and their interfaces.</p>
</li>
</ul>
<h2 dir="auto"><a id="user-content-business-context" class="anchor" aria-hidden="true" href="#business-context"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Business Context</h2>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">Specification of <strong>all</strong> communication partners (users, IT-systems, …)
with explanations of domain specific inputs and outputs or interfaces.
Optionally you can add domain specific formats or communication
protocols.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">All stakeholders should understand which data are exchanged with the
environment of the system.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">All kinds of diagrams that show the system as a black box and specify
the domain interfaces to communication partners.</p>
<p dir="auto">Alternatively (or additionally) you can use a table. The title of the
table is the name of your system, the three columns contain the name of
the communication partner, the inputs, and the outputs.</p>
<p dir="auto"><strong>&lt;Diagram or Table&gt;</strong></p>
<p dir="auto"><strong>&lt;optionally: Explanation of external domain interfaces&gt;</strong></p>
<h2 dir="auto"><a id="user-content-technical-context" class="anchor" aria-hidden="true" href="#technical-context"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Technical Context</h2>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">Technical interfaces (channels and transmission media) linking your
system to its environment. In addition a mapping of domain specific
input/output to the channels, i.e. an explanation with I/O uses which
channel.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Many stakeholders make architectural decision based on the technical
interfaces between the system and its context. Especially infrastructure
or hardware designers decide these technical interfaces.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">E.g. UML deployment diagram describing channels to neighboring systems,
together with a mapping table showing the relationships between channels
and input/output.</p>

<h1 dir="auto"><a id="user-content-solution-strategy" class="anchor" aria-hidden="true" href="#solution-strategy"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Solution Strategy</h1>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">A short summary and explanation of the fundamental decisions and
solution strategies, that shape the system’s architecture. These include</p>
<ul dir="auto">
<li>
<p dir="auto">technology decisions</p>
</li>
<li>
<p dir="auto">decisions about the top-level decomposition of the system, e.g.
usage of an architectural pattern or design pattern</p>
</li>
<li>
<p dir="auto">decisions on how to achieve key quality goals</p>
</li>
<li>
<p dir="auto">relevant organizational decisions, e.g. selecting a development
process or delegating certain tasks to third parties.</p>
</li>
</ul>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">These decisions form the cornerstones for your architecture. They are
the basis for many other detailed decisions or implementation rules.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">Keep the explanation of these key decisions short.</p>
<p dir="auto">Motivate what you have decided and why you decided that way, based upon
your problem statement, the quality goals and key constraints. Refer to
details in the following sections.</p>

<p dir="auto"><strong>Content.</strong></p>
<p dir="auto">The building block view shows the static decomposition of the system
into building blocks (modules, components, subsystems, classes,
interfaces, packages, libraries, frameworks, layers, partitions, tiers,
functions, macros, operations, datas structures, …) as well as their
dependencies (relationships, associations, …)</p>
<p dir="auto">This view is mandatory for every architecture documentation. In analogy
to a house this is the <em>floor plan</em>.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Maintain an overview of your source code by making its structure
understandable through abstraction.</p>
<p dir="auto">This allows you to communicate with your stakeholder on an abstract
level without disclosing implementation details.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">The building block view is a hierarchical collection of black boxes and
white boxes (see figure below) and their descriptions.</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/NetworkedAssets/arc42-in-markdown-template/blob/master/2.%20Single%20File/images/05_building_blocks-EN.png"><img src="/NetworkedAssets/arc42-in-markdown-template/raw/master/2.%20Single%20File/images/05_building_blocks-EN.png" alt="Hierarchy of building blocks" style="max-width: 100%;"></a></p>
<p dir="auto"><strong>Level 1</strong> is the white box description of the overall system together
with black box descriptions of all contained building blocks.</p>
<p dir="auto"><strong>Level 2</strong> zooms into some building blocks of level 1. Thus it contains
the white box description of selected building blocks of level 1,
together with black box descriptions of their internal building blocks.</p>
<p dir="auto"><strong>Level 3</strong> zooms into selected building blocks of level 2, and so on.</p>

<li>

</li>

<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">The runtime view describes concrete behavior and interactions of the
system’s building blocks in form of scenarios from the following areas:</p>
<ul dir="auto">
<li>
<p dir="auto">important use cases or features: how do building blocks execute
them?</p>
</li>
<li>
<p dir="auto">interactions at critical external interfaces: how do building blocks
cooperate with users and neighboring systems?</p>
</li>
<li>
<p dir="auto">operation and administration: launch, start-up, stop</p>
</li>
<li>
<p dir="auto">error and exception scenarios</p>
</li>
</ul>
<p dir="auto">Remark: The main criterion for the choice of possible scenarios
(sequences, workflows) is their <strong>architectural relevance</strong>. It is
<strong>not</strong> important to describe a large number of scenarios. You should
rather document a representative selection.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">You should understand how (instances of) building blocks of your system
perform their job and communicate at runtime. You will mainly capture
scenarios in your documentation to communicate your architecture to
stakeholders that are less willing or able to read and understand the
static models (building block view, deployment view).</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">There are many notations for describing scenarios, e.g.</p>
<ul dir="auto">
<li>
<p dir="auto">numbered list of steps (in natural language)</p>
</li>
<li>
<p dir="auto">activity diagrams or flow charts</p>
</li>
<li>
<p dir="auto">sequence diagrams</p>
</li>
<li>
<p dir="auto">BPMN or EPCs (event process chains)</p>
</li>
<li>
<p dir="auto">state machines</p>
</li>
<li>
<p dir="auto">…</p>
</li>
</ul>

<p dir="auto"><strong>Content.</strong></p>
<p dir="auto">The deployment view describes:</p>
<ol dir="auto">
<li>
<p dir="auto">the technical infrastructure used to execute your system, with
infrastructure elements like geographical locations, environments,
computers, processors, channels and net topologies as well as other
infrastructure elements and</p>
</li>
<li>
<p dir="auto">the mapping of (software) building blocks to that infrastructure
elements.</p>
</li>
</ol>
<p dir="auto">Often systems are executed in different environments, e.g. development
environment, test environment, production environment. In such cases you
should document all relevant environments.</p>
<p dir="auto">Especially document the deployment view when your software is executed
as distributed system with more then one computer, processor, server or
container or when you design and construct your own hardware processors
and chips.</p>
<p dir="auto">From a software perspective it is sufficient to capture those elements
of the infrastructure that are needed to show the deployment of your
building blocks. Hardware architects can go beyond that and describe the
infrastructure to any level of detail they need to capture.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Software does not run without hardware. This underlying infrastructure
can and will influence your system and/or some cross-cutting concepts.
Therefore, you need to know the infrastructure.</p>
<p dir="auto">Maybe the highest level deployment diagram is already contained in
section 3.2. as technical context with your own infrastructure as ONE
black box. In this section you will zoom into this black box using
additional deployment diagrams:</p>
<ul dir="auto">
<li>
<p dir="auto">UML offers deployment diagrams to express that view. Use it,
probably with nested diagrams, when your infrastructure is more
complex.</p>
</li>
<li>
<p dir="auto">When your (hardware) stakeholders prefer other kinds of diagrams
rather than the deployment diagram, let them use any kind that is
able to show nodes and channels of the infrastructure.</p>
</li>
</ul>

<h1 dir="auto"><a id="user-content-cross-cutting-concepts" class="anchor" aria-hidden="true" href="#cross-cutting-concepts"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Cross-cutting Concepts</h1>
<p dir="auto"><strong>Content.</strong></p>
<p dir="auto">This section describes overall, principal regulations and solution ideas
that are relevant in multiple parts (= cross-cutting) of your system.
Such concepts are often related to multiple building blocks. They can
include many different topics, such as</p>
<ul dir="auto">
<li>
<p dir="auto">domain models</p>
</li>
<li>
<p dir="auto">architecture patterns or design patterns</p>
</li>
<li>
<p dir="auto">rules for using specific technology</p>
</li>
<li>
<p dir="auto">principal, often technical decisions of overall decisions</p>
</li>
<li>
<p dir="auto">implementation rules</p>
</li>
</ul>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Concepts form the basis for <em>conceptual integrity</em> (consistency,
homogeneity) of the architecture. Thus, they are an important
contribution to achieve inner qualities of your system.</p>
<p dir="auto">Some of these concepts cannot be assigned to individual building blocks
(e.g. security or safety). This is the place in the template that we
provided for a cohesive specification of such concepts.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">The form can be varied:</p>
<ul dir="auto">
<li>
<p dir="auto">concept papers with any kind of structure</p>
</li>
<li>
<p dir="auto">cross-cutting model excerpts or scenarios using notations of the
architecture views</p>
</li>
<li>
<p dir="auto">sample implementations, especially for technical concepts</p>
</li>
<li>
<p dir="auto">reference to typical usage of standard frameworks (e.g. using
Hibernate for object/relational mapping)</p>
</li>
</ul>
<p dir="auto"><strong>Structure.</strong></p>
<p dir="auto">A potential (but not mandatory) structure for this section could be:</p>
<ul dir="auto">
<li>
<p dir="auto">Domain concepts</p>
</li>
<li>
<p dir="auto">User Experience concepts (UX)</p>
</li>
<li>
<p dir="auto">Safety and security concepts</p>
</li>
<li>
<p dir="auto">Architecture and design patterns</p>
</li>
<li>
<p dir="auto">"Under-the-hood"</p>
</li>
<li>
<p dir="auto">development concepts</p>
</li>
<li>
<p dir="auto">operational concepts</p>
</li>
</ul>
<p dir="auto">Note: it might be difficult to assign individual concepts to one
specific topic on this list.</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/NetworkedAssets/arc42-in-markdown-template/blob/master/2.%20Single%20File/images/08-Crosscutting-Concepts-Structure-EN.png"><img src="/NetworkedAssets/arc42-in-markdown-template/raw/master/2.%20Single%20File/images/08-Crosscutting-Concepts-Structure-EN.png" alt="Possible topics for crosscutting concepts" style="max-width: 100%;"></a></p>

<p dir="auto"><em>&lt;explanation&gt;</em></p>

<p dir="auto"><em>&lt;explanation&gt;</em></p>

<h2 dir="auto"><a id="user-content-concept-n" class="anchor" aria-hidden="true" href="#concept-n"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><em>&lt;Concept n&gt;</em></h2>

<h1 dir="auto"><a id="user-content-design-decisions" class="anchor" aria-hidden="true" href="#design-decisions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Design Decisions</h1>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">Important, expensive, large scale or risky architecture decisions
including rationals. With "decisions" we mean selecting one alternative
based on given criteria.</p>
<p dir="auto">Please use your judgement to decide whether an architectural decision
should be documented here in this central section or whether you better
document it locally (e.g. within the white box template of one building
block).</p>
<p dir="auto">Avoid redundancy. Refer to section 4, where you already captured the
most important decisions of your architecture.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Stakeholders of your system should be able to comprehend and retrace
your decisions.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">Various options:</p>
<ul dir="auto">
<li>
<p dir="auto">List or table, ordered by importance and consequences or:</p>
</li>
<li>
<p dir="auto">more detailed in form of separate sections per decision</p>
</li>
<li>
<p dir="auto">ADR (architecture decision record) for every important decision</p>
</li>
</ul>
<h1 dir="auto"><a id="user-content-quality-requirements" class="anchor" aria-hidden="true" href="#quality-requirements"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Quality Requirements</h1>
<p dir="auto"><strong>Content.</strong></p>
<p dir="auto">This section contains all quality requirements as quality tree with
scenarios. The most important ones have already been described in
section 1.2. (quality goals)</p>
<p dir="auto">Here you can also capture quality requirements with lesser priority,
which will not create high risks when they are not fully achieved.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Since quality requirements will have a lot of influence on architectural
decisions you should know for every stakeholder what is really important
to them, concrete and measurable.</p>
<h2 dir="auto"><a id="user-content-quality-tree" class="anchor" aria-hidden="true" href="#quality-tree"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Quality Tree</h2>
<p dir="auto"><strong>Content.</strong></p>
<p dir="auto">The quality tree (as defined in ATAM – Architecture Tradeoff Analysis
Method) with quality/evaluation scenarios as leafs.</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">The tree structure with priorities provides an overview for a sometimes
large number of quality requirements.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">The quality tree is a high-level overview of the quality goals and
requirements:</p>
<ul dir="auto">
<li>
<p dir="auto">tree-like refinement of the term "quality". Use "quality" or
"usefulness" as a root</p>
</li>
<li>
<p dir="auto">a mind map with quality categories as main branches</p>
</li>
</ul>
<p dir="auto">In any case the tree should include links to the scenarios of the
following section.</p>
<h2 dir="auto"><a id="user-content-quality-scenarios" class="anchor" aria-hidden="true" href="#quality-scenarios"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Quality Scenarios</h2>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">Concretization of (sometimes vague or implicit) quality requirements
using (quality) scenarios.</p>
<p dir="auto">These scenarios describe what should happen when a stimulus arrives at
the system.</p>
<p dir="auto">For architects, two kinds of scenarios are important:</p>
<ul dir="auto">
<li>
<p dir="auto">Usage scenarios (also called application scenarios or use case
scenarios) describe the system’s runtime reaction to a certain
stimulus. This also includes scenarios that describe the system’s
efficiency or performance. Example: The system reacts to a user’s
request within one second.</p>
</li>
<li>
<p dir="auto">Change scenarios describe a modification of the system or of its
immediate environment. Example: Additional functionality is
implemented or requirements for a quality attribute change.</p>
</li>
</ul>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">Scenarios make quality requirements concrete and allow to more easily
measure or decide whether they are fulfilled.</p>
<p dir="auto">Especially when you want to assess your architecture using methods like
ATAM you need to describe your quality goals (from section 1.2) more
precisely down to a level of scenarios that can be discussed and
evaluated.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">Tabular or free form text.</p>
<h1 dir="auto"><a id="user-content-risks-and-technical-debts" class="anchor" aria-hidden="true" href="#risks-and-technical-debts"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Risks and Technical Debts</h1>
<p dir="auto"><strong>Contents.</strong></p>
<p dir="auto">A list of identified technical risks or technical debts, ordered by
priority</p>
<p dir="auto"><strong>Motivation.</strong></p>
<p dir="auto">“Risk management is project management for grown-ups” (Tim Lister,
Atlantic Systems Guild.)</p>
<p dir="auto">This should be your motto for systematic detection and evaluation of
risks and technical debts in the architecture, which will be needed by
management stakeholders (e.g. project managers, product owners) as part
of the overall risk analysis and measurement planning.</p>
<p dir="auto"><strong>Form.</strong></p>
<p dir="auto">List of risks and/or technical debts, probably including suggested
measures to minimize, mitigate or avoid risks or reduce technical debts.</p>
